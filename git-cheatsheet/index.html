⚙️ Fundamentos y configuracion

Git es un sistema de control de versiones distribuido creado por Linus Torvalds. Antes de tocar un repositorio hay que identificarte: cada commit queda firmado con este nombre y email.

⚙️
Configuracion inicial
user.name, user.email, colores
▼
⚠️
Ojo: el email debe coincidir con el de tu cuenta de GitHub, o tus commits no se te atribuiran visualmente en el perfil.
Git — configuracion global
# Se aplica a todos los repositorios de este equipo git config --global user.name "Tu Nombre" git config --global user.email "tu@correo.com" # Colorea status, diff y log automaticamente git config --global color.ui auto # Ver toda la configuracion activa git config --list
📁
Crear e iniciar un repositorio
git init, la carpeta .git
▼
Git — init
# Crear directorio de trabajo mkdir mi-proyecto cd mi-proyecto # Inicializar el repositorio git init Inicializado repositorio Git vacio en ... # Iniciar directamente con la rama llamada "main" git init -b main # Renombrar la rama actual (ej: de master a main) git branch -M main
💡
git init crea una carpeta oculta .git con todo el historial y la configuracion del repositorio. Borrarla equivale a borrar el repositorio (los ficheros de trabajo sobreviven).
📦 Stage & snapshot

El ciclo diario de Git tiene tres zonas: working directory (lo que editas) → staging area (lo que preparas con add) → historial (lo que confirmas con commit).

📋
Ver el estado del repositorio
status, diff
▼
Git — status y diff
# Que archivos estan modificados / en staging / sin seguimiento git status # Diferencias de lo modificado pero AUN NO en staging git diff # Diferencias de lo que ya esta en staging pero sin commit git diff --staged # Historial de commits de la rama activa git log
✅
Anadir y confirmar cambios
add, commit
▼
Git — add + commit
# Preparar un archivo tal cual esta ahora mismo git add ejemplo.txt # Preparar TODOS los cambios del directorio actual git add . # Confirmar el contenido en staging como snapshot permanente git commit -m "He creado el fichero ejemplo.txt" # Atajo: add + commit en un paso (SOLO archivos YA rastreados) git commit -am "He modificado el fichero ejemplo.txt"
⚠️
Trampa comun: commit -am no funciona con archivos nuevos (sin rastrear). Para esos necesitas git add explicito al menos una vez.
✂️
Renombrar y eliminar ficheros
mv, rm, tracking de rutas
▼
Git — mv y rm
# Cambiar nombre y dejarlo preparado en staging git mv ejemplo.txt ejemplo2.txt git commit -am "He cambiado el nombre del fichero" # Eliminar del proyecto y preparar la eliminacion git rm ejemplo2.txt git commit -am "He borrado el fichero ejemplo2" # Ver el historial marcando archivos movidos/renombrados git log --stat -M
🌿 Ramas y fusion

Una rama es una linea independiente de desarrollo con su propio historial de commits, sin afectar a la rama principal hasta que decides fusionar.

🌿
Crear y moverte entre ramas
branch, checkout
▼
Git — branch y checkout
# Listar ramas locales (* marca la rama activa) git branch * main # Crear una rama nueva (NO te mueve a ella) git branch nueva-funcionalidad # Moverte a una rama existente git checkout nueva-funcionalidad # Truco: crear + moverte en un solo paso git checkout -b nueva-funcionalidad # Ver el ultimo commit de cada rama (para comprobar sincronizacion) git branch -v
💡
GitHub llama main a la rama principal desde 2020. En repositorios antiguos puede llamarse master — ambas cumplen el mismo rol.
🔄
Fusionar (merge) una rama
merge, eliminar rama fusionada
▼
Git — merge
# 1. Ponerte en la rama que va a RECIBIR los cambios git checkout main # 2. Fusionar la otra rama dentro de main git merge nueva-funcionalidad # 3. Ya fusionada, eliminar la rama local sobrante git branch -d nueva-funcionalidad
⚡
Rebase
historial lineal, uso avanzado
▼
Git — rebase
# Reaplica los commits de tu rama ENCIMA de [rama] git rebase main
⛔
Regla de oro: nunca hagas rebase de una rama que otras personas ya tienen clonada o sobre la que estan trabajando. Reescribe el historial y les rompe el suyo.
⚡ Conflictos de fusion

Aparecen cuando dos ramas modificaron la misma parte de un archivo. Git no puede decidir por ti cual version es la correcta.

⚠️
Leer y resolver un conflicto
marcas <<<<<<< = = = >>>>>>>
▼
Git — contenido con conflicto
cat prueba.txt <<<<<<< HEAD Hola como estas ======= hola que tal >>>>>>> nueva-rama
⚠️
Todo lo que hay entre <<<<<<< HEAD y ======= es tu version actual. Lo que sigue hasta >>>>>>> es la version entrante. Edita a mano, decide, borra las marcas y confirma.
Git — cerrar el conflicto
# Tras editar el archivo y quitar las marcas git add prueba.txt git commit -m "Resuelto conflicto en prueba.txt"
✅
Los conflictos por archivos anadidos o eliminados Git los resuelve solo. Los que duelen son los de lineas modificadas en ambas ramas.
🌐 Remoto y GitHub

GitHub guarda una copia de tu repositorio en un servidor remoto. Para no escribir contrasena en cada operacion se configura una clave SSH en tu perfil de GitHub (apartado SSH keys).

🔑
SSH vs HTTPS
que URL usar y cuando
▼
URLCuando usarla
git@github.com:usuario/repo.gitTus propios repositorios — permite push por SSH
https://github.com/usuario/repo.gitRepositorios ajenos que solo vas a clonar/leer
📁
Clonar y enlazar un repositorio
clone, remote add, push -u
▼
Git — clonar
# Descargar un repositorio completo con su historial git clone git@github.com:usuario/repo.git
Git — enlazar repositorio ya existente
# Crear repo vacio EN GITHUB (sin README) y luego: git remote add origin git@github.com:usuario/repo.git # Primer push: sube la rama y la enlaza (upstream) git push -u origin main # A partir de aqui, ya basta con: git push
⛔
Error clasico: si al crear el repo en GitHub marcas "Initialize with README", el remoto tendra commits que tu local no tiene y el primer push fallara. Creálo sin README si vas a subir un repo local ya existente.
🔁
Sincronizar: push, pull, fetch
subir, bajar, y bajar sin fusionar
▼
Git — sincronizacion
# Subir tus commits locales al remoto git push # Subir una rama nueva (las ramas NO se crean solas en GitHub) git push origin nombre-rama # Descargar y fusionar automaticamente (fetch + merge) git pull # Descargar SIN fusionar, para revisar antes de mezclar git fetch origin # Fusionar manualmente lo ya descargado git merge origin/main
↩️ Deshacer y viajar en el tiempo

Ordenados de menos a mas destructivo. Lee la descripcion entera antes de ejecutar cualquiera con --hard.

💾
Guardar cambios sin confirmar (stash)
seguro — reversible
▼
Git — stash
# Guardar temporalmente cambios sin confirmar git stash # Listar los stashes guardados (orden de pila) git stash list # Recuperar el ultimo y quitarlo de la pila git stash pop # Descartar el ultimo stash SIN aplicarlo git stash drop
↩️
Sacar del staging / volver a un commit
seguro — no pierde historial
▼
Git — reset suave y checkout
# Sacar del staging pero CONSERVAR los cambios en disco git reset ejemplo.txt # Viajar a un commit anterior sin perder los posteriores git checkout <id-del-commit> # Volver a donde estabas antes de moverte git switch -
⚠️
Reset --hard
destructivo — pierde cambios sin confirmar
▼
⛔
Irreversible en la practica: cualquier cambio sin confirmar en el momento de ejecutar esto se pierde. No hay "deshacer" facil.
Git — reset --hard
# Reescribe staging y working directory al estado de [commit] git reset --hard <commit> # Elimina PERMANENTEMENTE los ultimos 3 commits de la rama actual git reset --hard HEAD~3
🔍 Inspeccionar historial
📊
Comparar ramas y commits
log, diff entre ramas, show
▼
Git — comparaciones
# Commits que existen en branchA pero no en branchB git log branchB..branchA # Historial de un archivo, incluso a traves de renombres git log --follow ejemplo.txt # Diferencia de contenido entre dos ramas desde su origen comun git diff branchB...branchA # Ver un commit u objeto concreto en formato legible git show <SHA>
🚫
Ignorar archivos
.gitignore, exclusion global
▼
Git — core.excludesfile
# Patron de exclusion global para TODOS tus repositorios git config --global core.excludesfile ~/.gitignore_global
.gitignore — ejemplo de patrones
# Carpeta completa logs/ # Extension de archivo (comodin) *.notes # Cualquier carpeta que empiece por "pattern" pattern*/
✎️ Markdown

Lenguaje de marcas ligero que se convierte a HTML automaticamente. GitHub renderiza cualquier .md — por eso todo repo suele llevar un README.md describiendo el proyecto.

📝
Sintaxis basica
titulos, enfasis, listas, enlaces
▼
ElementoSintaxis
Titulo 1 / 2 / 3# Titulo / ## Titulo / ### Titulo
Negrita**bold text**
Cursiva*italicized text*
Cita> blockquote
Lista ordenada1. item / 2. item
Lista no ordenada* item
Codigo inline`code`
Enlace[title](https://example.com)
Imagen![alt text](image.jpg)
💡
Si abres el repositorio directamente en GitHub, los archivos .md se muestran ya renderizados como HTML, no como texto plano.
🔁 Pull Request y colaboracion

Un Pull Request (PR) es una peticion formal al dueno de un repositorio para que incorpore los cambios que propones. Es la forma estandar de colaborar en proyectos que no son tuyos.

🏆
Flujo completo de un PR
fork, clonar, rama, push, revision
▼
PasoAccion
1Fork del repositorio original desde GitHub (copia bajo tu cuenta)
2git clone git@github.com:tu-usuario/repo.git
3git checkout -b arregla-bug-x — nunca trabajes en main directamente
4git push origin arregla-bug-x — sube la rama a TU fork
5Abrir el Pull Request desde la interfaz de GitHub
6El propietario revisa y puede pedir ajustes
7Acepta (merge) o rechaza (se cierra sin fusionar)
🔄
Mantener tu fork actualizado
upstream, fetch, merge
▼
Git — sincronizar fork con el original
# 1. Posicionarte en la rama principal de tu fork git checkout main # 2. Anadir el remoto original (una sola vez) git remote add upstream git@github.com:autor-original/repo.git # 3. Traer los cambios del original SIN fusionar aun git fetch upstream # 4. Fusionarlos en tu main local git merge upstream/main
⚠️
Tu fork es una copia congelada en el tiempo. Si abres un PR desde un fork desactualizado es mucho mas probable que aparezcan conflictos que si lo sincronizas primero con upstream.
